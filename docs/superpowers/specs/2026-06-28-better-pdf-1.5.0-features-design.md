# better-pdf 1.5.0 features — design

Date: 2026-06-28

## Context

The web editor already targets `@ignaciano3/better-pdf` 1.5.0 and implements every
field/draw capability through 1.4.0. The 1.5.0 changelog adds exactly two things:

1. **Multi-select list boxes** — `addListBox(name, { multiSelect: true })` sets the
   choice Multiselect flag; the field reports `FieldInfo.multiSelect === true` and
   accepts `listBox.selectMultiple(values)`.
2. **Field flag / widget-visibility mutation on loaded fields** —
   `setReadOnly` / `setRequired` / `setExported` / `hide` / `show` / `setPrintable`
   / `setNoView`.

This spec covers **only #1 (multi-select list boxes)**. #2 has no builder creation
option (it exists only as setters on *loaded* fields) and would need a separate
post-build pass plus new UI; it is deliberately out of scope here.

## Library constraints that shape the design

- `addListBox` (the authoring path the editor uses) accepts a single `selected` and
  a single `defaultSelected` — no array form.
- `PdfListBox.selectMultiple(values)` sets `/V` (array) and `/I` (indices), and is
  only valid on a multi-select list box (throws `MultiSelectError` otherwise).
- There is **no** multi-value default API: `setDefaultSelected(value)` takes one
  value. So a multi-select reset-default stays single-valued.

## Design

### 1. Data model — `src/lib/pdf/types.ts` (`FieldElement`)

Two new optional props, meaningful for `field: 'listbox'` only:

- `multiSelect?: boolean` — sets the PDF Multiselect flag.
- `selectedValues?: string[]` — the set of initially-selected option values (the
  `/V` array). Used in place of the single `value` when `multiSelect` is on.

`defaultSelected` (single) is reused unchanged for the reset-default; multi-value
defaults are a documented non-goal (library limitation).

### 2. Export — `src/lib/pdf/build.ts`

- `authorField` listbox case passes `multiSelect: true` when set. Single-select
  list boxes are authored exactly as today.
- For a multi-select list box with one or more `selectedValues`, the builder
  authors the field **without** a single `selected`. A new post-build pass,
  `applyMultiSelections(bytes, selections)`, loads the built document, calls
  `form.getListBox(name).selectMultiple(validValues)` for each, and re-saves.
  - `validValues` = `selectedValues` filtered to entries present in the field's
    `options` (mirrors the existing defensive filtering for defaults).
  - `selections` is collected in `buildPdf` from `state.elements`
    (`field === 'listbox' && multiSelect && selectedValues?.length`).
  - Ordering in `buildPdf`: rebuild → `applyMultiSelections` (when any) →
    optional `flattenAllFields`. The multi-select pass must run before flatten so
    the selected appearance is baked when flattening.

### 3. Import round-trip — `src/routes/editor/extract-fields.ts`

`FieldInfoLike` already carries `multiSelect`. For a detected list box with
`info.multiSelect`:

- set `element.multiSelect = true`;
- split the comma-joined `info.value` into `selectedValues`, keeping only entries
  present in `info.options`; do **not** also set the single `element.value`.

(The library renders an array `/V` as a comma-joined string on read, so splitting
on `,` reverses it. Option values containing commas are a known best-effort edge.)

### 4. UI — `src/routes/editor/FieldPropertiesModal.svelte`

List-box only:

- A **Multi-select** checkbox bound to `field.multiSelect`. Turning it off clears
  `selectedValues`.
- When `multiSelect` is on, the single "Initial" `<select>` is replaced by a group
  of per-option checkboxes that add/remove the option from `selectedValues`.
  "Reset default" stays a single `<select>`.

### 5. Validation — `src/routes/editor/export-validate.ts`

In `validateElement`'s `field` case, accept:

- `multiSelect` — boolean when present;
- `selectedValues` — when present, an array of strings, length `≤ MAX_FIELD_OPTIONS`.

## Testing (TDD)

- **extract-fields**: a multi-select `FieldInfoLike` maps to an element with
  `multiSelect: true` and `selectedValues` filtered to valid options; single value
  is not set.
- **build**: a multi-select listbox element with `selectedValues` produces output
  whose reloaded `FieldInfo` reports `multiSelect === true` and a value covering the
  selected options; out-of-options values are dropped; flatten path still works.
- **export-validate**: accepts valid `multiSelect`/`selectedValues`; rejects a
  non-array `selectedValues` and an over-cap length.
- **FieldPropertiesModal**: toggling Multi-select swaps the initial selector and
  edits `selectedValues`; toggling off clears it.

## Out of scope

- Field flag / widget-visibility mutation (`hide`/`show`/`setPrintable`/`setNoView`/
  `setExported`) — no builder creation support; would need its own post-build pass.
- Multi-value reset-default — no library API.
